SCOUT-SECTIONS — VERIFICATION LOG
==================================

ENVIRONMENT
-----------
Hugo extended v0.123.7 (Ubuntu apt package). Theme min_version is
0.156, and the live workflow pins 0.157.0. The GitHub releases asset
CDN is not in this container's network allowlist, so the
production-pinned 0.157.0 build is the receiving end's responsibility.

Local rig workaround:
  The post-D7 partial uses `hugo.Data.scout_sections.items` (matching
  the existing partial style and theme convention). `hugo.Data` was
  introduced in Hugo 0.123.x and the rig hits a "can't evaluate field
  Data in type interface {}" error on the precise build of 0.123.7
  used here. For the rig only, the line was rewritten to
  `site.Data.scout_sections.items` — the older idiom that works on
  any Hugo ≥ 0.40. The deliverable's patch keeps `hugo.Data` matching
  the rest of the theme.

Verification rig is a minimal Hugo site against the post-D6 theme
tree plus all D7 changes:

  - 00-tokens.css, 01-reset.css (minimal), 02-layout.css (minimal),
    03-components-buttons.css, 05-components-sect-head.css (D5),
    10-section-hero.css (D5-patched), 15-section-join.css (D6),
    17-section-scout-sections.css (NEW D7), 41-feature-hero-intro.css
  - hero.html, section-header.html (D5), join.html (D6),
    scout-sections.html (NEW D7)
  - data/scout_sections.toml (D7-patched, .webp → .svg)
  - assets/images/sections/{squirrels,beavers,cubs,scouts,explorers,network}.svg
    (NEW D7 placeholders)

01-reset.css and 02-layout.css in the rig are minimal stand-ins for
the real theme's reset and layout — only enough to give `.container`
a max-width and `box-sizing: border-box` globally. The real theme
ships these properly via `assets/css/01-reset.css` and
`assets/css/02-layout.css`. Without the box-sizing reset, cards
render 19 px wider than expected because their `max-width: 380px`
is interpreted as content-box width instead of border-box.
